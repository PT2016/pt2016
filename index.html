Clean code conventions

"Always code as if the guy who ends up maintaining your code will be a violent psychopath who knows where you live. Code for readability."

Clean code can be achieved by

  • meaningful names (please see ch. 9 of the recommended reading material)
    • pronounceable, searchable, avoid encodings
    • intention-revealing names
  • language fundamentals (please see ch. 10.1 of the recommended reading material)
    • minimize the accessibility of classes and members (use encapsulation)
  • methods
    • small, easy to read & understand
    • do exactly one thing
    • keep number of arguments minimal & justified
  • objects & data (Law of Demeter https://en.wikipedia.org/wiki/Law_of_Demeter)
    • keep the variables private when you don't want anyone to depend on them
    • hide implementation, expose interface
    • "talk to friends, not strangers":
      • a method f of class C should only call methods of:
        • C
        • objects created by f
        • objects passed as an argument of f
        • objects held in an instance of C
      • a method should NOT invoke methods on objects that are returned by any allowed functions
  • formatting (please see ch 4, 6 and 7 of the recommended reading material)
  • comments (please see ch. 5 of the recommended reading material)
    • explain your intend, but avoid noisy or redundant comments
  • exceptions & error handling
    • have a clear separation between error handling and business logic (define a normal flow and the exceptional flows)
    • avoid returning/passing null in case of exceptions (difficult to handle) - thrown an exception instead
    • provide context (informative messages) with exceptions

Clean code should obey the following principles:

  • single responsibility & high cohesion
    • specialized classes/methods
    • keep modules/elements together
  • low coupling
    • minimize dependencies

Recommended reading: https://oracle.com/technetwork/java/codeconventions-150003.pdf
